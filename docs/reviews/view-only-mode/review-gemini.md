# View-Only Mode Code Review

- **[HIGH]** `app/(wallet)/wallet/deposits/deposits-page-client.tsx:209`: Missed short-circuit guard on deposit creation. The `onCreate` callback triggers `createDeposit.mutate(input)` directly. A user could bypass the disabled UI button and hit the backend guard. **Fix**: Add `if (viewOnly) { toast.error(walletCopy.viewOnlyDepositDisabled); return; }` inside `onCreate` before the mutation.
- **[HIGH]** `app/(wallet)/wallet/settings/page.tsx:254`: Missed guard on the optimize wallet trigger. The `handleOptimize` function calls `optimizeWallet.mutate` unconditionally, and the "Optimize Now" button does not disable itself when view-only. **Fix**: Import and call `useWalletViewOnly()`, add a `viewOnly` short-circuit return in `handleOptimize`, and add `|| viewOnly` to the button's `disabled` expression.
- **[MEDIUM]** `app/(wallet)/wallet/settings/page.tsx:316`: The UX prompt "Optimization can be attempted..." is still shown for view-only wallets. The spec explicitly required suppressing any optimize prompts. **Fix**: Change the ternary condition to `optimizationNeeded && !viewOnly`.
- **[MEDIUM]** `app/(wallet)/wallet/export/page.tsx:93`: The export dialog shows an empty spend key when exporting a view-only wallet, but it does not include the copy-only note specified by the spec. **Fix**: Import `useWalletViewOnly()` and conditionally render "Spend key is blank — view-only wallet" when `viewOnly` is true.
- **[LOW]** `lib/services/mock/wallet.service.ts:62`: Mock/Real divergence on QR imports. The mock state sets `mockViewOnly` only if `input.method === "keys"`. A user importing a view-only QR code in mock mode will get a spendable (false) wallet state, whereas real mode correctly reads the QR payload. **Fix**: While adhering strictly to the spec, this is a divergence; consider adding a heuristic to check if the QR payload implies watch-only in mock mode.
